# Marketing Journeys + Client Creation Fields — Design

**Date:** 2026-05-20
**Project:** ASTRAL — internal agency platform
**Scope:** Third sub-project. Earlier shipped: A (CRM for inbound leads), B (Sales coaching + Tasks per client). Out of scope here: home overview dashboard, global UI polish pass, "add mission" UI rework, sales-coaching lifecycle metrics — each becomes its own future sub-project.

## Goal

Capture a marketing path (organic / paid / both) per client at creation time, drive each path through a sequential journey of stages with per-video checklists where applicable, and mirror every active stage as a row in the existing per-client Tasks list so Missions and Journeys stay in sync.

Also add a `salesMeetingsTarget` field on `Client` so the Sales workspace can display "held / target" instead of just held.

## Non-Goals (v1)

- Changing `videoCount` after a journey is created (delete + recreate).
- Archiving an unfinished journey (delete is the only removal).
- Per-video filming dates (a single shared `filmingDate` per filming stage).
- Per-video doc links (a single shared `docLink` per stage).
- AI-assisted next-step suggestions.
- Notifications when a stage unlocks or filming day arrives.
- A unified "all clients progress" view (separate future sub-project).

## Information Architecture

A new tab **"מסלולים"** in the marketing workspace sub-nav, inserted **before** "משימות":

```
דשבורד | דף נחיתה | חומרים | ניתוח AI | קמפיינים | מסלולים | משימות
```

Path: `/clients/[id]/marketing/journeys`.

The Tasks tab is unchanged structurally; journey-spawned tasks appear in it with a colored badge (`אורגני` / `ממומן`) to distinguish them from free-form tasks.

## Data Model

### `Client` (modify)

Add:
```prisma
salesMeetingsTarget Int?
```

Existing relations (`meetings`, `tasks`, …) stay. Add:
```prisma
journeys Journey[]
```

### `Journey`

| Field | Type | Notes |
|---|---|---|
| `id` | String | PK |
| `clientId` | String | FK → Client, cascade |
| `kind` | String | `organic` or `paid` |
| `videoCount` | Int | ≥1 (planned videos/ads) |
| `status` | String | `active` or `completed` |
| `currentStageIndex` | Int | 0..N-1 |
| `createdAt` | DateTime | |
| `updatedAt` | DateTime | |

Constraints / indexes:
- `@@unique([clientId, kind])` — at most one journey of each kind per client.
- `@@index([clientId])`.

### `JourneyStage`

| Field | Type | Notes |
|---|---|---|
| `id` | String | PK |
| `journeyId` | String | FK → Journey, cascade |
| `index` | Int | 0,1,2,3 — position in the journey |
| `kind` | String | `writing` / `filming` / `editing` / `strategy` / `ads_writing` / `ad_filming` / `creative` |
| `mode` | String | `single` or `per_video` |
| `docLink` | String? | optional URL (scripts doc, finished-videos folder, ad strategy doc, etc.) |
| `filmingDate` | DateTime? | only meaningful for `filming` / `ad_filming` stages |
| `status` | String | `locked` / `active` / `done` |
| `taskId` | String? | FK → Task; set when the stage enters `active` and a Task is auto-spawned |

Indexes: `@@index([journeyId])`, `@@index([taskId])`.

### `JourneyVideoItem`

| Field | Type | Notes |
|---|---|---|
| `id` | String | PK |
| `stageId` | String | FK → JourneyStage, cascade |
| `index` | Int | 1..videoCount |
| `done` | Boolean | default `false` |
| `doneAt` | DateTime? | set when toggled to true, cleared when toggled back |

Created in bulk **when the parent stage transitions from `locked` → `active`** (not at journey creation), so we lazily materialize per-video rows only for stages the user actually reaches. Exception: if `videoCount` is changed, the only path is journey deletion + recreation — see Non-Goals.

Index: `@@unique([stageId, index])`.

### `Task` (modify)

Add:
```prisma
linkedStageId String? @unique
```

`@unique` because the journey↔task mapping is 1:1. Add an index implicitly via `@unique`. Add the reverse on `JourneyStage` as `task Task?` already declared above.

## Progression Rules

1. **Creation.** Creating a journey writes all `JourneyStage` rows up-front, all `locked` except index 0 which is set `active`. The `active` stage spawns its `Task` and its per-video items (if `mode = per_video`). `journey.currentStageIndex = 0`.

2. **Marking a single-mode stage done.** Direct toggle: button "סמן בוצע" on the stage. Pre-condition for `filming`/`ad_filming`: `filmingDate` is set AND `filmingDate <= now()`. Otherwise the button is disabled with a tooltip "תאריך הצילום עדיין לא הגיע".

3. **Marking a per_video stage done.** Cannot be marked done directly. It becomes done automatically when ALL its `JourneyVideoItem.done = true`. Toggling the last item triggers stage completion in the same transaction.

4. **Stage completion side-effects** (in a single Prisma transaction):
   - Stage `status = done`.
   - Linked `Task.status = done`, `Task.completedAt = now()`.
   - If a next stage exists: it transitions `locked → active`, spawns its Task, and (if `per_video`) creates its `JourneyVideoItem` rows. `Journey.currentStageIndex` increments.
   - If no next stage: `Journey.status = completed`.

5. **Reschedule filming.** "דחה לתאריך אחר" sets a new `filmingDate`. No status change. UI confirms the change.

6. **Reverting Task done → open.** If user un-checks a journey-spawned Task in the Missions tab, the linked Stage flips back from `done` to `active`, and any subsequent stages that were already `active` due to chain progression also revert. Concretely: walk forward from the reverted stage and set every later stage back to `locked` (and delete their `JourneyVideoItem` rows + their linked Tasks). `Journey.status` returns to `active` if it was `completed`. `currentStageIndex` is reset to the reverted stage's index.

   This is intentionally aggressive: un-completing an earlier stage means later work is now stale. Confirmation prompt in the UI: "ביטול ההשלמה ימחק שלבים שכבר נפתחו אחרי השלב הזה. להמשיך?"

7. **Toggling a per-video item from done back to open** while the parent stage is `done` reverts the stage to `active` and cascades per rule 6 (because the chain has advanced past it).

## Journey ↔ Task Sync

**Forward (stage drives task):**
- Stage spawn → create `Task { space: "marketing", title: stageTaskTitle, description: "שלב {n+1} מתוך {N} במסלול {kindLabel}", priority: "normal", clientId, linkedStageId: stage.id }`. Set `stage.taskId = task.id`.
- Stage `done` → patch Task to `done` (same transaction as completion in rule 4).
- Stage cascaded back to `locked` (rule 6 forward walk) → delete the Task entirely (it shouldn't be on the missions tab anymore because the stage isn't active).

**Backward (task drives stage):**
- PATCH `/api/tasks/[id]` checks for `linkedStageId`. If set and `status` is changing:
  - `open → done`: invoke stage completion (rule 4).
  - `done → open`: invoke stage revert (rule 6).
- PATCH on linked tasks for any field other than `status` works normally.
- DELETE on a linked task is forbidden (409) — user must delete via the Journeys tab.

**Free-form tasks** (no `linkedStageId`) behave exactly as today: full CRUD, no journey side-effects.

## API

| Method | Path | Purpose |
|---|---|---|
| POST | `/api/clients` (modify) | Accept extended body: `{ name, description?, salesMeetingsTarget?, journeys?: [{kind, videoCount}, …] }`. Creates Client + journeys in one transaction. |
| PATCH | `/api/clients/[id]` (modify) | Accept `salesMeetingsTarget` (nullable int). |
| POST | `/api/clients/[id]/journeys` | Add a journey to an existing client. Body: `{ kind, videoCount }`. 409 if a journey of that kind already exists. |
| DELETE | `/api/journeys/[id]` | Delete journey + cascade stages, video items, linked tasks. |
| PATCH | `/api/journeys/[id]/stages/[stageId]` | Update stage. Body: `{ docLink?, filmingDate?, markDone? }`. `markDone: true` triggers rule 4 with pre-conditions. Only allowed on `active` stages; for `per_video`, `markDone` is rejected (must use video items). |
| PATCH | `/api/journeys/[id]/stages/[stageId]/videos/[videoIndex]` | Body: `{ done: boolean }`. Triggers rule 4 if it completes the stage, or rule 6 if it un-completes a previously-done stage. |
| PATCH | `/api/tasks/[id]` (modify) | If `linkedStageId` is set on the row, status changes route through journey advance/revert logic. |
| DELETE | `/api/tasks/[id]` (modify) | Reject (409 `"linked to journey stage"`) if `linkedStageId` is set. |

All routes: `runtime = "nodejs"`, Zod validation, no auth.

## UI

### Client creation — 2-step modal

`src/app/clients/new-client-form.tsx` is rewritten as a button that opens a modal.

**Step 1:**
- שם (required)
- תיאור (optional, textarea)
- יעד פגישות מכירה (number, ≥0, optional — blank = no target)

**Step 2:**
- Two checkboxes: "מסלול אורגני" / "מסלול ממומן" (either, both, or neither selectable)
- For each checked: a number input "כמות סרטונים מתוכננת" / "כמות מודעות מתוכננת", required ≥1
- "סיום" button — POSTs everything in one request.

The user can leave Step 2 with both unchecked (a Client with no journey) and add later via the per-client UI.

### `/clients/[id]/marketing/journeys` page

**Empty state** (no journey exists): centered card with two large buttons:
- "+ הוסף מסלול אורגני"
- "+ הוסף מסלול ממומן"

Click → dialog: "כמות סרטונים מתוכננת" → POST → refresh.

**Populated:** one card per journey (organic first, paid second). Each card:
- Header: "מסלול אורגני · 12 סרטונים" + delete button (× with confirm).
- Horizontal stepper of stages. Visual states:
  - `locked` — gray, non-interactive
  - `active` — accent border, clickable
  - `done` — green check
- A progress bar below the stepper (% of stages done).
- A panel below the stepper showing the **active** stage's details:
  - **single (non-filming)**: `docLink` input (text, autosaved on blur) + "סמן בוצע" button.
  - **filming/ad_filming**: date picker (autosaved) + "סמן בוצע" (disabled with tooltip if date is in future) + "דחה לתאריך אחר" link that opens an inline date picker.
  - **per_video**: `docLink` input + grid of checkboxes labeled "סרטון 1", "סרטון 2", …, "סרטון N". Each toggles instantly. A "X/N הושלמו" counter beside the grid.
- If `journey.status = "completed"`: a green banner above the stepper — "סיימת את כל המשימות ללקוח".

If a kind is missing for this client, a "+ הוסף מסלול {kind}" button appears below the existing cards.

The marketing tasks page now needs the `journey.kind` of each linked stage to render the badge. The server component eager-loads `stage` and `stage.journey` for tasks with `linkedStageId`.

## Validation Rules

- `videoCount` ≥ 1 enforced at API + UI.
- `salesMeetingsTarget` if provided must be a non-negative integer (`null` to clear).
- `docLink` if provided must parse as `URL` (Zod `.url()`), else 400.
- `filmingDate` accepted as ISO datetime string.
- Stage `markDone` rejected when `stage.mode === "per_video"` (400 "use video items").
- Stage `markDone` rejected when `stage.status !== "active"` (409).
- Stage `markDone` on filming/ad_filming rejected when `filmingDate` is null or in the future (409).
- Reverting an already-`active` stage is a no-op (200).

## Out of Scope (explicit)

- Editing `videoCount` after creation.
- Per-video doc links or per-video filming dates.
- Tasks that link to per-video items (only stage-level mapping).
- Drag-reorder of stages.
- Multiple journeys of the same kind per client.
- Notifications/reminders.
- Home overview dashboard (separate sub-project).
- Cross-client journey progress aggregation (separate sub-project).
- Global UI polish pass (separate sub-project).
- AddMission UI improvements beyond the journey badges (separate sub-project).
